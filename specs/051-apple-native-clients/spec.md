# Feature Specification: Native Apple Clients (iOS, macOS, watchOS SDUI Targets)

**Feature Branch**: `051-apple-native-clients`

**Created**: 2026-07-06

**Status**: Draft

**Input**: User description: "Create an iOS, macOS, and watchOS client for the backend orchestrator system. iOS should mirror the Android UI and flows, macOS should mirror the Windows client UI and flows. For the watch UI, since this is a small screen, if the user is not logged in it needs to display a login QR code generated by the backend; the user scans it with any other of their devices (phone, tablet, website) to log in on the watch. The orchestrator and ROTE must gracefully degrade components to the watch, and on the watch the user should be able to start a new conversation, use voice commands, and receive UI plus text-to-speech output."

## Overview

AstralDeep currently ships three user-facing clients — the server-rendered web shell, the native
Windows desktop client (PySide6), and the native Android SDUI client (Kotlin/Compose) — all
consuming one server-driven UI contract: **astralprims defines → orchestrator renders → ROTE
adapts** (Constitution II), with cross-client behavior pinned by the committed protocol manifest
(`backend/shared/ui_protocol.json`: 47 push frame types, 67 accepted actions, 35 component
types) and its per-client drift guards (feature 044, Constitution XII).

This feature adds the Apple family as three new first-party SDUI targets:

- **iOS (iPhone + iPad)** — the platform twin of the Android client (041/044): same screens,
  flows, streaming behavior, attachment handling, chrome surfaces, and reconnect contract,
  expressed natively in SwiftUI.
- **macOS** — the platform twin of the Windows client: same window anatomy (top bar, chat rail,
  component canvas), same desktop feature rows from the 044 parity matrix (attachments, table
  pagination, live theme restyle, image/plotly rendering, bounded-queue reconnect).
- **watchOS** — the smallest screen the product has ever targeted, and the proof case for ROTE
  graceful degradation. Because credential entry on a watch is impractical, an unauthenticated
  watch shows a **backend-generated QR code** (plus a short human-readable code) that the user
  scans or types from any other device to sign the watch in — an OAuth 2.0 Device Authorization
  Grant (RFC 8628) brokered by the backend against the existing Keycloak posture. Once signed
  in, the watch can start a conversation, take voice-dictated input, and deliver every response
  both as degraded visual components and as spoken output via on-device text-to-speech driven by
  the server's existing spoken-rendition (`voice`) render target.

The backend already carries significant head starts this feature exercises rather than invents:
a `watch` ROTE device profile (1 column, 120-char text cap, no charts/tables/code), a registered
`voice` render target that collapses component trees to speakable text, per-socket ROTE
adaptation on every fanout path, the additive `POST /api/auth/logout` with per-client revocation
(`auth_revocation_queue.client_id`, 044), and the named-device-profile pattern used to add
`android` in 041. Server-side deltas are additive and small; **zero new backend third-party
runtime dependencies** and **zero third-party dependencies in the Swift clients**
(Constitution V).

## Clarifications

### Session 2026-07-06

- Q: What mechanism should power the watch QR sign-in? → A: **Keycloak device grant
  (RFC 8628), proxied by the backend** — the backend requests the device authorization from the
  IdP, generates and serves the QR (verification URI + code) itself, and the watch polls the
  backend for completion. No bespoke approval protocol; revocation and session semantics stay
  IdP-native (Constitution VII posture).
- Q: How does a user complete watch sign-in from the website (no camera)? → A: **Short-code
  entry** — the watch displays the QR *and* a short human-readable code; phones/tablets scan the
  QR with any camera, while a browser user types the short code at the verification page. No
  in-browser webcam QR scanner is required by this feature.
- Q: Where do speech-to-text and text-to-speech run? → A: **On-device Apple frameworks** —
  system dictation / Speech framework for input, AVSpeechSynthesizer-class TTS for output. The
  server supplies the *what to speak* (its existing `voice` spoken-rendition target); the watch
  supplies the *how*. Zero new backend dependencies, no audio streamed over the wire.
- Q: Third-party dependency posture for the Swift clients? → A: **Zero third-party** — Apple
  frameworks only (URLSession WebSocket, ASWebAuthenticationSession + hand-rolled PKCE exactly
  as the Windows client hand-rolls it over stdlib, Codable, CoreImage/Swift Charts). The new
  Apple toolchain itself is recorded for Constitution V lead-dev approval; its third-party
  dependency set is empty.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - The daily chat loop on iPhone and iPad, indistinguishable from Android (Priority: P1)

A researcher who already uses AstralDeep on Android picks up an iPhone or iPad and gets the same
product: sign in through the system browser, land on their chat history, open or start a chat,
send a message (with slash-command typeahead and attachments), watch progress while agents work,
see rich components render natively and update in place, manage agents and permissions from the
chrome menu, switch theme, and sign out knowing the server session is actually gone.

**Why this priority**: iOS is the largest missing audience and the most complete flow set to
mirror; it exercises the entire existing contract (auth, chat, streaming, canvas, chrome,
attachments) and is independently shippable as an MVP on its own.

**Independent Test**: Build and run the iOS app against the dev backend; execute the 044 US1
"daily chat loop" script end to end on an iPhone simulator and an iPad simulator; compare every
step's disposition against the Android column of the parity matrix.

**Acceptance Scenarios**:

1. **Given** a signed-out iOS app, **When** the user signs in via the system browser (PKCE) with
   a role-bearing account, **Then** the app lands on the chat list, the WebSocket registers with
   an `ios` device profile, and a token lacking `user`/`admin` roles is refused with the same
   no-access experience the other clients give.
2. **Given** an open chat, **When** the user sends a message with an attachment, **Then** the
   staged chip, upload, `parser_status` surfacing, acknowledgement, progress signals, streamed
   narrative, and final components all appear natively, in the same order and meaning as on
   Android.
3. **Given** a canvas with components, **When** the server pushes `ui_upsert` ops or a
   `component_action` result (e.g. table pagination), **Then** the affected components update in
   place without losing scroll position.
4. **Given** a dropped connection, **When** the network returns, **Then** the client reconnects
   with the shared backoff contract (1 s base, ×2, 30 s cap), flushes its bounded outbound queue
   (64 frames, FIFO, drop-oldest with a visible signal), and resumes the session silently
   (`resumed: true`).
5. **Given** a server `error` or `stream_error` frame tied to a user action, **When** it
   arrives, **Then** the user visibly learns about it (banner/toast) — never a silent drop.
6. **Given** the user signs out, **When** logout completes, **Then** the server-side session and
   refresh token are revoked (`POST /api/auth/logout` with this client's id), local tokens are
   wiped, and the next launch requires interactive sign-in.

### User Story 2 - A macOS desktop client with everything the Windows client does (Priority: P1)

A desktop user on a Mac gets the Windows client experience natively: a single window with the
top bar (identity, connection status, new-chat, search), the left chat rail with history, and
the component canvas; loopback/system-session PKCE sign-in; attachments from the file dialog or
drag-and-drop; paginated tables; live theme restyle; images and charts rendered natively; and
the same dependable reconnect/queue behavior.

**Why this priority**: macOS is the second half of "desktop parity" and reuses the entire shared
Swift foundation built for iOS; it is independently testable against the Windows column of the
parity matrix and shippable on its own.

**Independent Test**: Run the macOS app on the development Mac against the dev backend; execute
the Windows-client verification script from 044 step for step; every row the Windows client
passes must pass on macOS (or carry a documented, device-appropriate variant).

**Acceptance Scenarios**:

1. **Given** a signed-out macOS app, **When** the user signs in, **Then** an RFC 8252-style
   system-browser PKCE flow completes against the Keycloak realm using a dedicated allow-listed
   public client, and tokens live in the platform keychain.
2. **Given** the main window, **When** a chat is opened from the rail, **Then** the transcript
   re-hydrates (including server-rendered component history), and new turns stream into the
   canvas exactly as on Windows.
3. **Given** a component-bearing response with a table larger than one page, **When** the user
   pages it, **Then** `component_action`-driven pagination round-trips and updates the table in
   place.
4. **Given** a theme change from settings, **When** it is applied, **Then** the running window
   restyles live (no restart), mirroring the Windows restyle behavior.
5. **Given** files attached via dialog or drag-and-drop, **When** the message sends, **Then**
   staged chips, `parser_status`, and delivery behave identically to the Windows client.

### User Story 3 - Signing in on the watch by pointing another device at it (Priority: P2)

A user straps on their watch, opens AstralDeep, and — having never typed a password on a watch —
sees a QR code and a short code with an expiry countdown. They point their phone camera at it
(any camera app), land on the sign-in page, authenticate (or are already signed in), and
approve. Within seconds the watch flips to the signed-in home screen. At a desktop browser
instead, they type the short code at the verification page and the same thing happens.

**Why this priority**: It is the gate to every other watch capability, but it depends on nothing
from US1/US2 and is independently demonstrable end to end with just the backend and a watch
simulator.

**Independent Test**: With the feature enabled and the realm's device grant configured, cold
launch the watch app signed out; complete sign-in once by scanning with a phone camera and once
by typing the short code in a desktop browser; verify expiry rotation, denial, and reuse
attempts behave as specified.

**Acceptance Scenarios**:

1. **Given** an unauthenticated watch app, **When** it launches, **Then** it displays a
   backend-generated QR code, the short human-readable code, and time remaining — with no
   credential entry affordance anywhere on the watch.
2. **Given** the displayed QR, **When** the user scans it with any phone/tablet camera and
   completes sign-in and approval on that device, **Then** the watch transitions to signed-in
   state within one poll interval, with a session equivalent to an interactive login (roles
   gate enforced; 365-day interactive anchor set at approval time).
3. **Given** a desktop browser with no camera, **When** the user visits the verification page
   and types the short code, **Then** the watch signs in identically.
4. **Given** an unapproved code reaching its TTL, **When** it expires, **Then** the watch
   automatically fetches and displays a fresh QR/code before the stale one is presented as
   valid, and the expired code can never produce a session.
5. **Given** a user who denies the approval, or a token whose account lacks the `user`/`admin`
   role, **When** the watch polls, **Then** it shows a friendly failure state with a retry
   affordance, and no session or partial session exists server-side.
6. **Given** a signed-in watch, **When** the user signs out from the watch, **Then** the server
   revokes the session/refresh token (offline-tolerant, attributed to the watch client id),
   local tokens are wiped, and the watch returns to the QR screen.

### User Story 4 - A hands-free conversation on the wrist (Priority: P2)

A signed-in watch user taps "New conversation", dictates a question by voice, and gets the
answer both ways at once: compact watch-fitted components on screen and the response read aloud
through on-device text-to-speech. They can replay or stop the speech, scroll the visual answer
with the crown, and pick up any recent conversation from a short list.

**Why this priority**: This is the watch's core value — it turns the platform's existing SDUI
and spoken-rendition machinery into a genuinely usable wearable client. It requires US3 to sign
in but is otherwise independent.

**Independent Test**: On a signed-in watch (simulator or device), start a new conversation by
voice, receive a multi-component answer, and verify both the visual rendering and that spoken
playback begins promptly; repeat from the recent-conversations list.

**Acceptance Scenarios**:

1. **Given** the signed-in watch home, **When** the user taps once, **Then** a new conversation
   opens with dictation ready (scribble/keyboard available as fallback).
2. **Given** a dictated message, **When** it sends, **Then** it flows through the standard chat
   pipeline (permissions, PHI gate, audit) exactly like a typed message from any other client —
   no privileged shortcut.
3. **Given** a response with narrative and components, **When** it arrives, **Then** the watch
   renders the ROTE watch-profile adaptation on screen and begins speaking the server-supplied
   spoken rendition via on-device TTS, honoring the system silent/Do-Not-Disturb state.
4. **Given** spoken playback, **When** the user taps stop or replay, **Then** playback obeys
   immediately; navigating away stops speech.
5. **Given** the watch home, **When** the user opens the recents list, **Then** a bounded list
   of their most recent conversations (titles) opens the selected conversation with its watch-
   adapted transcript.

### User Story 5 - Every component the server can send degrades gracefully to the watch (Priority: P2)

Whatever an agent produces — tables, charts, code, heroes, timelines, file cards — the watch
never shows a blank canvas, a crash, or a mystery. Each component type renders in a compact
watch-legible form or degrades along a deterministic fallback chain to something readable, with
a clear "continue on another device" affordance when the full experience needs a bigger screen.

**Why this priority**: This is the explicit product guarantee that makes the watch trustworthy,
and it hardens the ROTE degradation ladder for every future constrained target. It is testable
headlessly against the manifest without any of the other stories.

**Independent Test**: Drive a sweep that pushes every one of the 35 manifest component types
(and representative nested layouts) through a watch-profile socket; assert every delivery yields
non-empty, watch-bounded output and a non-empty spoken rendition, with zero errors.

**Acceptance Scenarios**:

1. **Given** each of the 35 component types in the manifest, **When** it is delivered to a
   watch-profile socket, **Then** ROTE produces a non-empty, size-bounded adaptation (per the
   watch profile's caps) or its documented fallback — never an empty canvas or unhandled type.
2. **Given** a component whose interactivity exceeds the watch profile (action count, input
   complexity), **When** it renders, **Then** the watch shows the read-only summary plus a
   "continue on phone or desktop" affordance instead of broken controls.
3. **Given** a narrative longer than the watch text cap, **When** it renders, **Then** the
   visual is truncated with a clear indication the full response exists on other devices, while
   the spoken rendition covers the response within the voice target's documented bounds.
4. **Given** a turn carrying attachments, **When** it reaches the watch, **Then** attachment
   names appear as read-only chips (no upload affordance exists on the watch).

### User Story 6 - Evidence and drift guards that keep six clients honest (Priority: P3)

A maintainer can point to one place — the parity matrix and the committed protocol manifest —
and know exactly what every client does with every frame and component type. CI fails if any
Apple client's vocabulary drifts from the manifest, and the verification bundle proves with
legible captures that all three new clients were exercised live.

**Why this priority**: Process armor, not user-facing value — but it is what kept 041/044
honest, and the client count doubling makes it more valuable, not less.

**Independent Test**: Change a manifest entry locally and observe each Apple drift guard fail;
review the recorded verification bundle for completeness and legibility.

**Acceptance Scenarios**:

1. **Given** the 044 parity matrix, **When** this feature lands, **Then** it carries explicit
   per-frame and per-component dispositions for iOS, macOS, and watchOS alongside web, Windows,
   and Android.
2. **Given** a vocabulary change to `ui_protocol.json`, **When** CI runs, **Then** the Swift
   drift-guard tests fail until the Apple clients' dispositions are updated — same contract as
   the Windows and Android guards.
3. **Given** the feature's verification bundle, **When** reviewed, **Then** it contains legible
   captures of the live chat loop on iOS, macOS, and watchOS against the dev backend.

### Edge Cases

- QR/short code approved twice, or replayed after expiry or after the watch already signed in:
  every code is single-use; late/duplicate approvals produce a clear "code no longer valid" and
  never a second session.
- Watch clock skew vs. code TTL: expiry decisions are server-authoritative; the countdown is
  advisory only.
- IdP unreachable or device grant not enabled on the realm: the sign-in start endpoint refuses
  cleanly with an actionable message on the watch (bounded retry), mirroring the web login's
  IdP-unreachable posture; nothing fails open.
- Polling abuse: poll respects the server-issued interval and `slow_down`; the backend
  rate-limits start/poll per device and address.
- Approval from a device signed into a *different* account than intended: the watch session
  belongs to whichever account approved — the signed-in home must display the account identity
  so a mistake is immediately visible, and sign-out is one tap away.
- Token refresh fails on any Apple client (revoked/hard-cap): client receives `auth_required`,
  wipes local tokens, and lands on its sign-in surface (QR screen on watch) — same semantics as
  web/Windows/Android.
- Connection drops mid-stream on the watch: partial narrative remains visible; on reconnect the
  transcript re-hydrates; TTS never speaks the same turn twice.
- Dictation yields empty/garbled text: the watch lets the user review before send (confirm
  affordance) rather than auto-sending silently.
- TTS interrupted by wrist-down/screen sleep or an incoming call: playback pauses/stops per
  platform convention; replay is available.
- Very large `ui_render` payloads on the watch: ROTE caps sizes server-side (existing profile
  bounds); the watch additionally guards against oversized frames without crashing.
- Unknown future frame types or component types reaching any Apple client: safely ignored
  (frames) or rendered via the documented fallback (components) — forward compatibility, no
  crash.
- iPad multitasking sizes (Split View/Slide Over): the iOS client reports viewport changes via
  the existing device-update path so ROTE can re-derive layout, as the Android client does on
  fold/rotation.
- macOS window closed vs. app quit: WS teardown mirrors the Windows client's close frames;
  device-aware chrome error/close semantics (044) apply unchanged.
- The same user simultaneously active on watch, iPhone, Mac, and web: existing per-socket ROTE
  adaptation and workspace fanout already handle mixed-profile delivery; the watch socket must
  never receive an unadapted payload.

## Requirements *(mandatory)*

### Functional Requirements

**Shared Apple foundation**

- **FR-001**: The feature MUST deliver three native Apple app targets (iOS iPhone+iPad, macOS,
  watchOS) sharing a first-party protocol/core layer (frame models, WS client with the shared
  backoff/queue contract, auth/session management, component-model parsing), with **no
  third-party dependencies** in any Apple target and no web views for product surfaces (the
  system authentication browser session is the sole exception).
- **FR-002**: Each client MUST register over the existing WS contract reporting its device
  identity (`ios`, `macos`, `watch`) with honest capability fields (viewport, touch, microphone)
  and `supported_types` so ROTE adapts per socket; iOS MUST report viewport changes (rotation,
  iPad multitasking) through the existing device-update action.
- **FR-003**: Every push frame type in the committed manifest MUST have an explicit disposition
  (handle / deliberately ignore) on each Apple client, recorded in the parity matrix; unknown
  frame types MUST be ignored safely.
- **FR-004**: Every component type in the committed manifest MUST render natively on iOS and
  macOS per their profiles, and render-or-degrade on watchOS per FR-032; no client may crash or
  blank on any manifest type.
- **FR-005**: All three clients MUST implement the shared reconnect contract (1 s base backoff,
  ×2, 30 s cap, reset on success) and the bounded 64-frame outbound queue with FIFO flush and a
  user-visible dropped-frame signal, matching the Windows/Android contract.
- **FR-006**: All wire changes introduced by this feature MUST be additive and
  backward-compatible; `backend/shared/ui_protocol.json` remains the single vocabulary source
  and gains any additions before client work consumes them.
- **FR-007**: iOS and macOS MUST authenticate via OIDC PKCE through a system browser session
  against the existing realm using dedicated public client ids that are allow-listed for the
  orchestrator (`KEYCLOAK_ALLOWED_AZP`); tokens MUST live in the platform keychain; silent
  refresh MUST NOT extend the 365-day interactive anchor; `auth_required` MUST route the user
  to interactive sign-in.
- **FR-008**: Sign-out on every Apple client MUST call the server-revoking logout with the
  client's id (offline-tolerant revocation queue attribution) and wipe local tokens; a signed-
  out client MUST hold no usable credential material.
- **FR-009**: The feature MUST introduce **zero new backend third-party runtime dependencies**
  (QR generation included) and MUST record the new Apple client toolchain (with its empty
  third-party set) for Constitution V approval.
- **FR-010**: All Apple-client traffic MUST flow through the existing chat pipeline and its
  gates (tool permissions, PHI gate, audit, concurrency caps) with no native-only bypass; audit
  rows MUST be attributable to the originating client.

**iOS client (Android twin)**

- **FR-011**: The iOS client MUST mirror the Android screen set and flows: sign-in →
  chat history (paginated list with open/delete) → chat canvas; message input with
  slash-command typeahead; attachments via document/photo pickers with staged chips,
  per-file upload, removal, and `parser_status` surfacing; native chrome menu (agents,
  settings, theme); agent/permission management surfaces; silent resume with the `resumed`
  register flag.
- **FR-012**: The iOS client MUST present the full streaming lifecycle: send acknowledgement,
  progress/status signals while agents work, incremental narrative streaming with terminal
  semantics, and visible stream-error handling with retry.
- **FR-013**: The iOS client MUST apply in-place canvas updates (upsert ops: replace by
  component identity, remove, append) preserving scroll position, and MUST round-trip
  component actions (including table pagination) through the standard action path.
- **FR-014**: The iOS client MUST adapt between iPhone and iPad presentations using the
  server-driven profile plus reported viewport — matching the Android client's phone/tablet
  behavior rather than inventing a separate iPad information architecture.
- **FR-015**: Server `error` frames tied to user actions MUST be visibly surfaced (banner or
  toast) on iOS; silent drops are defects.

**macOS client (Windows twin)**

- **FR-016**: The macOS client MUST mirror the Windows window anatomy: top bar (identity,
  connection status, new-chat, search), left chat rail (history with selection and context
  actions), and the native component canvas.
- **FR-017**: The macOS client MUST match the Windows feature rows of the parity matrix:
  attachments via file dialog and drag-and-drop with `parser_status` chips; table pagination;
  live theme restyle without restart; native image rendering; chart rendering for the chart
  component family (rasterized rendering acceptable, as on Windows); reconnect/backoff with
  visible connection status.
- **FR-018**: The macOS client MUST re-hydrate transcripts and workspace component history on
  chat load equivalently to the Windows client.
- **FR-019**: The macOS client MUST persist session state across app restarts with silent
  refresh (within the interactive anchor), mirroring Windows behavior.

**watchOS sign-in (backend-brokered device grant)**

- **FR-020**: An unauthenticated watch MUST present a sign-in screen containing a
  backend-generated QR code, the short human-readable code, and expiry feedback — and MUST NOT
  offer credential entry on the watch.
- **FR-021**: The backend MUST expose device sign-in endpoints brokering the IdP's RFC 8628
  device authorization grant: a *start* operation returning the QR (backend-rendered via
  first-party code) plus its encoded verification URI, the short code, expiry, poll interval,
  and an opaque poll handle; and a *poll* operation returning pending / slow-down / approved
  (session material) / denied / expired. The watch MUST NOT contact the IdP directly.
- **FR-022**: The QR MUST encode the complete verification URI such that scanning with any
  stock camera app on any device opens a browser page where the user authenticates and
  approves; the short code MUST be enterable at the verification page to serve camera-less
  devices (desktop browser path). No in-browser webcam scanning is required.
- **FR-023**: Codes MUST be single-use and TTL-bound; the watch MUST auto-rotate to a fresh
  code before expiry while displayed; polling MUST respect the server interval and slow-down;
  the backend MUST rate-limit start and poll per client address; expired/denied/used codes
  MUST never yield a session.
- **FR-024**: Approval MUST produce a watch session with interactive-login semantics: the
  365-day interactive anchor set at approval, identical role gating (a roleless account gets a
  friendly refusal on the watch and no session, with IdP-side grant revoked), and identical
  refresh/hard-cap behavior thereafter.
- **FR-025**: The device sign-in lifecycle MUST be audited (auth class: started, approved,
  denied, expired) with correlated identifiers and the originating client id; token material
  MUST never be logged.
- **FR-026**: Device sign-in MUST be feature-flagged (`FF_DEVICE_LOGIN`, default on) and MUST
  fail closed: flag off, IdP unreachable, or device grant not enabled on the realm each yield a
  clean refusal and an actionable watch message — never a hung or partially-authenticated
  state.
- **FR-027**: Watch sign-out MUST revoke server-side (attributed to the watch client id via the
  existing offline-tolerant revocation queue), wipe local tokens, and return to the QR screen.

**watchOS conversation, voice, and speech**

- **FR-028**: A signed-in watch MUST offer: one-tap new conversation from the root screen, a
  bounded recent-conversations list (most recent titles) that opens a watch-adapted transcript,
  and visible account identity with one-tap sign-out.
- **FR-029**: Watch message input MUST support system voice dictation as the primary method
  (with scribble/keyboard fallback where the platform provides it), with user confirmation
  before send; dictated messages flow through the standard chat path and gates unchanged.
- **FR-030**: For watch-profile sockets, the orchestrator MUST attach a spoken rendition of
  each delivered turn (produced by the existing `voice` render target) as an additive field on
  the existing delivery frames; the watch MUST speak it via on-device TTS on arrival, honoring
  system silent/DND state, with stop and replay controls; navigation away stops playback; a
  turn is never auto-spoken twice.
- **FR-031**: ROTE watch adaptation MUST happen server-side before send on every delivery path
  (initial render, upserts, re-hydration, fanout to mixed-profile sockets); the watch client
  MUST never receive web HTML or an unadapted component payload.

**ROTE degradation guarantees (watch)**

- **FR-032**: Every component type in the manifest MUST map, under the watch profile, to either
  a compact native rendering or a deterministic documented fallback chain terminating in
  readable text — zero blank canvases, zero unhandled types, zero crashes; the existing watch
  profile bounds (columns, text/table caps) are the authoritative limits.
- **FR-033**: Components whose interactivity exceeds the watch profile MUST degrade to a
  read-only summary plus a "continue on another device" affordance; narrative beyond the text
  cap MUST indicate that the full response is available on other clients; turns carrying
  attachments MUST show name-only chips (no watch upload affordance).

**Backend and platform deltas**

- **FR-034**: The backend MUST add named full-capability device profiles for `ios` and `macos`
  (the 041 `android` pattern) and reuse the existing `watch` profile; `supported_types`
  negotiation MUST be honored for all three.
- **FR-035**: No schema change is anticipated; device sign-in state MUST be held statelessly
  (opaque encrypted handle to the watch) or in-memory. If persistence proves necessary during
  planning, it ships as an idempotent guarded startup migration with documented rollback
  (Constitution IX).
- **FR-036**: Operator documentation MUST be updated: realm settings (device grant enablement,
  new public client ids, azp allow-list entries) and production deployment notes; the protocol
  manifest MUST document any additive fields introduced (e.g. the spoken-rendition field).

**Verification and CI**

- **FR-037**: The 044 parity matrix MUST be extended with iOS, macOS, and watchOS columns
  covering every push frame type and component type with explicit dispositions.
- **FR-038**: The shared Swift layer MUST include manifest drift guards equivalent to the
  Windows and Android guards, failing when the committed manifest's vocabulary changes without
  a client disposition update.
- **FR-039**: CI MUST gain an Apple job (macOS runner) that builds all three app targets and
  runs the Swift test suites (including drift guards) on every PR, alongside — not replacing —
  every existing gate (Constitution XI).
- **FR-040**: The feature MUST capture a live verification bundle with legible evidence of the
  chat loop on all three Apple clients against the dev backend, following the 044 evidence
  conventions.
- **FR-041**: Interactive elements on iOS and macOS MUST carry accessibility labels
  (VoiceOver); the watch UI MUST respect system text-size settings within the profile's
  layout bounds.

### Key Entities

- **Device Sign-In Request**: One watch sign-in attempt — short code, encoded verification URI,
  opaque poll handle, TTL/expiry, poll interval, and state (pending → approved | denied |
  expired), single-use; brokered against the IdP, surfaced by the backend.
- **Apple Client Identity**: The per-platform public client ids (iOS, macOS, watch) — azp
  allow-list entries, revocation-queue attribution, and audit attribution.
- **Spoken Rendition**: The voice-target output attached additively to watch-bound deliveries;
  bounded speakable text derived from the same components the watch renders visually.
- **Device Profile (ios, macos)**: New named full-capability ROTE profiles joining `windows`
  and `android`; `watch` reused as-is.
- **Parity Matrix Extension**: The 044 matrix grown to six client columns; the authoritative
  per-frame/per-component disposition record.
- **Shared Apple Core**: The first-party Swift protocol/core layer (frame models, WS client,
  auth/session, renderer registry) consumed by all three app targets.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user with no existing watch session completes QR sign-in from cold launch —
  scan with a phone camera, approve, watch signed in — in under 60 seconds excluding credential
  typing, in 10 out of 10 consecutive attempts; the short-code (browser) path succeeds in the
  same bound.
- **SC-002**: 100% of the manifest's 35 component types delivered to a watch-profile socket
  yield non-empty, size-bounded visual output and a non-empty spoken rendition in an automated
  sweep — zero blanks, zero errors, zero unhandled types.
- **SC-003**: 100% of the manifest's 47 push frame types have explicit recorded dispositions
  for each of iOS, macOS, and watchOS; a manifest vocabulary change fails CI on every Apple
  drift guard until dispositions are updated.
- **SC-004**: iOS passes every parity-matrix row the Android client passes, and macOS passes
  every row the Windows client passes (identical disposition or a documented
  device-appropriate variant) — zero unexplained regressions in the matrix.
- **SC-005**: On the watch, spoken playback of a completed response begins within 2 seconds of
  the turn's final frame (p95 over 20 consecutive turns).
- **SC-006**: A voice-dictated round trip (start dictation → confirm send → spoken reply
  begins) requires no more than two deliberate taps beyond speaking.
- **SC-007**: Dependency audit: zero new third-party runtime packages on the backend and zero
  third-party packages in any Apple target, verified by manifest/lockfile inspection in CI.
- **SC-008**: 100% of tested sign-outs from each Apple client leave the server session unusable
  (subsequent WS registration refused / `auth_required`) with the revocation attributed to the
  correct client id.
- **SC-009**: 0 of N replay attempts against used, expired, or denied sign-in codes produce a
  session (N ≥ 20 across states); poll traffic never exceeds the server-issued interval under
  sustained observation.
- **SC-010**: All pre-existing CI gates remain green, and the new Apple CI job runs on every PR
  with the drift guards active.

## Assumptions

- **Single feature, three targets**: One spec/branch covers all three clients plus the small
  backend deltas, mirroring how 044 treated multiple clients; user stories remain independently
  implementable slices (iOS alone is a viable MVP).
- **IdP capability**: The Keycloak deployment supports the OAuth 2.0 Device Authorization Grant
  and the operator will enable it for the watch's public client; verified against the dev realm
  during planning (the grant is a per-client toggle in supported Keycloak versions). If the
  deployed version lacks it, upgrading the realm is in scope for planning, not a spec change.
- **Client identities**: Per-platform public client ids (working names `astral-ios`,
  `astral-macos`, `astral-watch`) following the `astral-desktop`/`astral-mobile` precedent;
  exact naming settled at plan time; all appended to the azp allow-list.
- **Minimum OS versions**: iOS/iPadOS 17, macOS 14, watchOS 10 — current-generation SwiftUI
  baselines; the watch app is an independent watchOS app (functions without the paired phone
  present, over Wi-Fi/LTE, network permitting).
- **Toolchain**: Xcode on the development Mac (present) and macOS runners available to CI
  (GitHub-hosted); code signing uses development certificates — App Store distribution,
  notarization pipelines, and MDM are out of scope.
- **Backend reachability**: Devices/simulators reach the dev backend over the local network or
  tunnels; TLS posture for non-local testing follows the existing production deployment docs.
- **Spoken rendition source**: The existing `voice` render target's output bounds (list/table/
  timeline caps) are the accepted speech bounds for the watch; no new speech synthesis service
  is introduced server-side.
- **Web client unchanged**: The browser completes watch sign-in at the IdP's hosted
  verification page via short-code entry; no webcam QR scanner and no ES5 render-layer changes
  are required by this feature.
- **Out of scope**: push notifications, watch complications, Siri intents/App Shortcuts,
  visionOS/CarPlay, offline message queuing beyond the bounded reconnect queue, attachment
  upload from the watch, account self-registration, backend speech-to-text, and any change to
  agent/tool behavior.

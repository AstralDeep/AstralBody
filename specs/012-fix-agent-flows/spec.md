# Feature Specification: Fix Agent Creation, Test, and Management Flows

**Feature Branch**: `012-fix-agent-flows`
**Created**: 2026-05-01
**Status**: Draft
**Input**: User description: "As a user, I want to be able to create and test a new agent through the UI. Fix the Agent creation flow to get to the agent test screen. Have the draft agent actually start running and responding to the user. Have the draft agent, on approval, get added to the list of live agents. The Agent management UI is opening but when I click on the agent to see the permissions the modal closes and refreshes the page."

## Clarifications

### Session 2026-05-01

- Q: When the user tests a draft on the Test screen, what does the draft have access to? → A: Real credentials via the Permissions screen — the user must supply any required credentials before testing tools that need them; tools that don't require credentials still work.
- Q: How are drafts approved — is there an admin reviewer step? → A: No admin reviewer step. After the user approves the draft, the system runs the existing automated security checks; if they pass, the draft auto-promotes to live. If they fail, the draft is rejected back to the user with the failure details. There is no "pending review" state in this feature.
- Q: How long is an unpromoted draft retained? → A: Kept indefinitely unless the creator deletes it.
- Q: What concrete latency target applies to a draft response on the Test screen? → A: Full response within 60 seconds (95th percentile); no separate first-token target.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Reach the Test Screen After Creating an Agent (Priority: P1)

A user opens the Create Agent flow, fills in the agent's name, description, and any tools, and triggers generation. After generation succeeds, the experience must transition into a Test screen where the user can interact with the draft. Today this transition is broken — the user fills out the form and triggers generation, but never lands on a usable test surface.

**Why this priority**: Without reaching the test screen, every other capability in this feature (testing the draft, approving it, promoting it to live) is unreachable. This is the single largest gap blocking the create-and-test loop the user described.

**Independent Test**: A user can complete the creation steps end-to-end, click the action that should advance them to testing, and visually confirm they are now on a Test screen with a chat input that targets the draft they just created — without any other story being implemented.

**Acceptance Scenarios**:

1. **Given** a user has filled in a valid agent name, description, and at least one tool, **When** they finish the review step and trigger generation, **Then** they are taken to a Test screen for that draft agent within a few seconds of generation completing.
2. **Given** generation of a draft has completed successfully, **When** the user is on the Test screen, **Then** the screen clearly identifies which draft is loaded, shows its current status, and presents a chat input ready for the user to send a message.
3. **Given** generation fails, **When** the user is shown the failure, **Then** the user receives a clear explanation and at least one next action (retry, edit definition, or close) instead of being stranded on a blank or frozen view.

---

### User Story 2 - Draft Agent Actually Runs and Responds (Priority: P1)

Once on the Test screen, the user sends a message to the draft agent and expects it to behave like a real agent — receive the message, run, and respond. Today, the draft does not actually start or respond, so the test screen has no functional value.

**Why this priority**: Testing is the entire reason the test screen exists. Without a runnable, responsive draft, the user cannot evaluate whether the agent they generated is correct before promoting it. P1 alongside Story 1 because together they form the minimum viable create-and-test loop.

**Independent Test**: From the Test screen, a user types a prompt the draft agent should be able to handle, sends it, and receives a real response generated by the draft within a reasonable response window — without the agent having been approved or moved to the live list.

**Acceptance Scenarios**:

1. **Given** the user is on the Test screen for a freshly generated draft, **When** they send their first message, **Then** the draft starts (if not already running) and the full response from the draft is rendered in the chat within 60 seconds.
2. **Given** the user has had at least one exchange with the draft, **When** they send a follow-up message, **Then** the draft responds in the same conversation context without requiring a manual restart.
3. **Given** the draft fails to start or crashes during a turn, **When** the user is awaiting a response, **Then** the user is shown a clear error explaining the draft is not responding and is offered at least one recovery action (retry, refine, or abandon).
4. **Given** a draft is being tested, **When** the user closes the Test screen and returns to the same draft later, **Then** they can resume testing the same draft without losing the draft's identity or having to regenerate.

---

### User Story 3 - Approving a Draft Promotes It to the Live Agents List (Priority: P1)

After testing, the user approves the draft. On approval, the draft must appear in the user's list of live agents and become usable from the same places any other live agent is usable. Today, approval does not reliably result in the agent showing up as live.

**Why this priority**: This is the payoff for the entire create-and-test journey. If approval doesn't promote the agent to live, users have no way to actually use what they built. P1 because the user explicitly asked for it and the loop is incomplete without it.

**Independent Test**: A user with a draft on the Test screen approves it, navigates to the live agents list, and sees the new agent listed there and selectable, without needing a page reload or manual restart.

**Acceptance Scenarios**:

1. **Given** the user has a draft loaded on the Test screen, **When** they approve it and approval succeeds, **Then** the draft transitions to a live state and appears in the live agents list visible to that user.
2. **Given** an approved agent is now live, **When** the user opens it from the live agents list, **Then** they can interact with it the same way as any other live agent without re-approving or re-generating.
3. **Given** the user approves a draft, **When** the system runs the existing automated security checks, **Then** the draft is auto-promoted to live if the checks pass and is rejected back to the user with specific failure details if they do not — there is no human-in-the-loop review step in between.
4. **Given** the security checks fail and the draft is rejected, **When** the failure occurs, **Then** the draft remains in the drafts list in a rejected state, the user is shown which checks failed and why, and the user can refine and re-submit the same draft for approval.

---

### User Story 4 - Viewing Agent Permissions Without the Page Refreshing (Priority: P1)

In the Agent Management UI, the user clicks an agent to view or change its permissions. The Permissions screen must open and stay open. Today, the modal opens but immediately closes and the page reloads, making it impossible to view or edit any permissions.

**Why this priority**: Permissions are the user's only way to control what their agents can do. With this bug, no user can configure any agent permission through the UI. P1 because it is a complete blocker for an existing surface area, separate from the create/test loop but equally broken.

**Independent Test**: A user with at least one agent opens the Agent Management UI, clicks on the agent, and the Permissions screen opens and remains open with the agent's current scopes visible — no page reload, no automatic close.

**Acceptance Scenarios**:

1. **Given** an agent is visible in the Agent Management UI, **When** the user clicks it to view permissions, **Then** the Permissions screen opens and remains open until the user explicitly dismisses it.
2. **Given** the Permissions screen is open, **When** the user toggles a scope, confirms a credential entry, or interacts with any control inside the screen, **Then** the page does not reload and focus stays inside the Permissions screen.
3. **Given** the user dismisses the Permissions screen normally (close button, backdrop click, or escape), **When** they return to the Agent Management UI, **Then** the agent list is unchanged and any scope changes they saved are reflected.

---

### Edge Cases

- The user starts generating a draft, then closes the Create Agent flow before generation completes — does the draft persist as an in-progress draft, get discarded, or get marked as failed? Behavior must be consistent and visible to the user.
- The user reaches the Test screen but never sends a message — the draft should be allowed to stop consuming runtime resources (i.e., the draft's running process can wind down) while the draft record itself is preserved.
- The user approves a draft that another tab/session already approved — the second approval should be a no-op rather than producing a duplicate live agent or an error.
- The user clicks an agent in the Management UI while the agent is mid-update from another session — the Permissions screen should still open and display the most recent state, not crash on stale data.
- The user opens the Permissions screen, edits a scope, and loses network connectivity before saving — the user must be told the change did not persist rather than seeing it appear saved.
- An approved draft fails to start as a live agent (post-promotion runtime failure) — it must not silently disappear; the user must be able to see that the live agent exists but is unhealthy and take action.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST advance a user from the Create Agent review step to a Test screen for the newly generated draft once generation completes successfully.
- **FR-002**: System MUST identify, on the Test screen, which draft is loaded (name and current status) so the user always knows what they are testing.
- **FR-003**: System MUST start the draft agent on demand when the user begins testing it, without requiring the user to perform any separate "start" action.
- **FR-004**: System MUST deliver the draft agent's response to the user's test messages in the same conversation, with the full response arriving within 60 seconds (95th percentile of test turns).
- **FR-005**: System MUST surface a clear, recoverable error state when generation, draft startup, or a draft response fails, and offer at least one explicit next action.
- **FR-006**: System MUST allow a user to leave the Test screen and return to the same draft later without losing the draft or its identity.
- **FR-006a**: A draft agent under test MUST run against the same credential model as live agents — it uses the user's real credentials supplied through the Permissions screen. Tools whose required credentials have not been provided MUST be disabled or clearly marked as unavailable on the Test screen, while tools that do not require credentials remain usable.
- **FR-006b**: System MUST surface, from the Test screen, which (if any) credentials the draft is missing and provide a path for the user to supply them via the Permissions screen without losing the in-progress test session.
- **FR-007**: Users MUST be able to approve a draft from the Test screen.
- **FR-008**: System MUST, on successful approval, promote the draft into the user's live agents list so the agent appears alongside other live agents the user owns.
- **FR-009**: System MUST make an approved agent usable from the same surfaces as any other live agent immediately after promotion, without a manual page reload.
- **FR-010**: When the user approves a draft, System MUST run the existing automated security checks and decide the outcome with no human reviewer step: passing checks promote the draft to live; failing checks leave the draft in a rejected state with the specific failures shown to the user.
- **FR-010a**: A rejected draft MUST remain editable and re-submittable by its owner — refining the draft and approving again MUST re-run the security checks against the updated draft.
- **FR-011**: System MUST prevent duplicate live agents being created when the same draft is approved more than once.
- **FR-012**: The Agent Management UI MUST allow the user to open an agent's Permissions screen without triggering a page reload or automatically dismissing the screen.
- **FR-013**: Interactions inside the Permissions screen (toggling scopes, entering credentials, confirming dialogs) MUST NOT cause the page to reload or the Permissions screen to close unexpectedly.
- **FR-014**: System MUST persist permission changes only when the user explicitly saves them and MUST notify the user if a save fails.
- **FR-015**: System MUST keep an unhealthy live agent visible in the live agents list with a status that communicates its health, rather than silently removing it.
- **FR-016**: System MUST retain a draft indefinitely until its owner explicitly deletes it; idle or never-tested drafts MUST NOT be auto-expired.
- **FR-017**: Users MUST be able to delete their own drafts, and a deleted draft MUST be removed from the drafts list.

### Key Entities *(include if feature involves data)*

- **Draft Agent**: A user-created agent that has been defined and (optionally) generated but not yet promoted to live. Carries a status reflecting its lifecycle stage (being defined, generating, ready to test, rejected, approved/live). Owned by the user who created it. Rejected drafts can be refined and re-approved by their owner.
- **Live Agent**: An agent that is approved and available for use from the user's live agents list. Has the same identity that the originating Draft Agent had.
- **Permission Scope**: A named bundle of capabilities (e.g., reading tools, writing tools, system access) that a user can grant to or revoke from an agent. An agent has zero or more scopes enabled at any time.
- **Agent Credential**: A piece of secret information an agent needs in order to be fully usable (e.g., an external API key). Tied to a specific agent and supplied through the Permissions screen.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: At least 95% of users who start a new agent and successfully complete the definition steps reach the Test screen for the resulting draft.
- **SC-002**: At least 95% of test messages sent to a freshly generated draft receive a complete draft response within 60 seconds.
- **SC-003**: 100% of drafts that the user approves and that pass the automated security checks are visible in the live agents list within 10 seconds of the checks completing, without a manual page reload.
- **SC-004**: 100% of attempts to open an agent's Permissions screen result in the screen opening and staying open until the user dismisses it.
- **SC-005**: Zero unintended page reloads occur when interacting with the Agent Management UI or the Permissions screen across the full set of supported user actions.
- **SC-006**: A new user, given an agent idea, can create, test, approve, and use a working live agent in a single session without contacting support.

## Assumptions

- The existing Create Agent multi-step form (define → tools → review → generate) is the intended flow; this feature fixes the broken transition into testing rather than redesigning the form.
- The existing draft-vs-live distinction in the system is the intended model; a draft is a private work-in-progress and a live agent is one approved for use. This feature does not introduce a new lifecycle stage.
- The existing automated security checks are the only gate between user approval and the draft going live; no human reviewer step is introduced. Their pass/fail criteria are kept as-is — this feature only ensures the resulting promotion-or-rejection behavior is reliable and visible.
- "Live agents list" refers to the user's view of agents available to them, not a global registry visible to all users; existing visibility/ownership rules are preserved.
- The Permissions screen's existing scope model (read/write/search/system-style scopes plus per-agent credentials) is preserved; this feature fixes the open/close/refresh bug and does not redesign the permissions concept.
- A draft under test uses the same credential mechanism as a live agent (user-supplied credentials via the Permissions screen). No separate sandbox or mock-tool layer is introduced for drafts.
- The 60-second response budget for a draft response covers the entire turn (including any tool calls); there is no separate first-token / streaming-start target in this feature.

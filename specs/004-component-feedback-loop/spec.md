# Feature Specification: Component Feedback & Tool Auto-Improvement Loop

**Feature Branch**: `004-component-feedback-loop`
**Created**: 2026-04-28
**Status**: Draft
**Input**: User description: "As a user, I want to be able to provide feedback on the output of components. As a developer, I want the system to take this feedback and auto update tools that continue to produce errors."

## Clarifications

### Session 2026-04-28

- Q: Rolling window length and underperforming-tool threshold? → A: 14-day rolling window; eligibility = ≥ 25 dispatches in window; flag if failure-rate ≥ 20% OR negative-feedback-rate ≥ 30%.
- Q: Where does the prompt-injection / nefarious-behavior screen run in the pipeline? → A: Both — inline at submit time AND as a pre-pass inside the improvement loop, defense in depth.
- Q: Rate limiting / spam protection on feedback submission? → A: Per-user-per-component dedup window (10 s default); subsequent submissions within the window supersede the prior record without writing a new audit row.
- Q: Retention window for user retract / amend? → A: 24 hours from the original submission timestamp.
- Q: How does an admin learn that a new tool has been flagged underperforming? → A: Passive in-app indicator (badge with pending-flag count on the admin review surface) plus an audit-log `tool_flagged` event on each first transition into the underperforming state. No email / push / Slack.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Provide feedback on a rendered component (Priority: P1)

A user is interacting with the AstralBody dashboard and notices that a component the system just rendered is wrong, irrelevant, broken, or unhelpful. They want a fast way to tell the system this so the experience improves over time. Without leaving the dashboard, they tap a feedback control directly on the offending component, indicate whether the output was good or bad, optionally pick a category that captures *what* was wrong, and optionally add a sentence of detail. They get a clear acknowledgement that their feedback was received.

**Why this priority**: This is the user-facing half of the feature. Without an in-product way to capture feedback, no improvement signal exists. This story can ship and deliver value (richer audit/quality data) even if the auto-improvement half (US-2) is not yet built.

**Independent Test**: Submit feedback on a rendered component as a regular user, confirm the feedback is recorded against the correct component dispatch and the correct user, and confirm the user sees an acknowledgement within one second.

**Acceptance Scenarios**:

1. **Given** a component has just rendered in the user's dashboard, **When** the user activates the feedback control on that component and selects "thumbs down" with category "wrong-data" and a comment, **Then** the system records the feedback tied to the originating tool dispatch, the user, the conversation, and the moment in time, and the user sees an in-place confirmation.
2. **Given** a component has just rendered, **When** the user submits feedback with no category and no comment (sentiment only), **Then** the feedback is still recorded with category "unspecified" and an empty comment.
3. **Given** the user has already submitted feedback for a specific component, **When** they submit feedback again for the same component, **Then** the latest feedback supersedes the previous entry for that component dispatch (the prior entry is preserved as history but no longer counts toward quality signals).
4. **Given** the user is logged in as a regular user, **When** they attempt to view feedback submitted by another user, **Then** the system rejects the request and returns no other-user feedback.
5. **Given** the user submits feedback, **When** the network is briefly unavailable, **Then** the feedback control surfaces a retry affordance and the feedback is delivered once connectivity returns, with no silent loss.

---

### User Story 2 - Surface tools that consistently produce errors and propose improvements (Priority: P1)

An admin (the "developer" in the user request) opens an admin-only review surface and sees a ranked list of tools whose recent quality has fallen below a configured threshold. For each underperforming tool, the system shows: the rate of failed dispatches over a rolling window, the rate of negative user feedback, the most common failure modes drawn from category counts and quarantined-text-free aggregates, and a proposed improvement generated by the system's existing tool-improvement loop. The admin reviews the proposal, then accepts, rejects, or edits-and-accepts it. Their decision is recorded for audit.

**Why this priority**: This is the developer-facing half. It closes the loop and is what makes user feedback actionable. Without it, feedback piles up unused. P1 because the user request explicitly couples both halves; without US-2 the user request is not satisfied.

**Independent Test**: Seed a tool with sufficient simulated failures + negative feedback to cross the threshold; confirm the tool appears in the admin review surface within one evaluation cycle; confirm a remediation proposal is generated and presented; confirm an admin can accept it and the acceptance is audit-logged with their identity.

**Acceptance Scenarios**:

1. **Given** a tool's combined failure-and-negative-feedback rate has crossed the underperforming threshold over the rolling window, **When** the next evaluation cycle runs, **Then** the tool appears in the admin review queue tagged "underperforming" with the supporting evidence visible.
2. **Given** an underperforming tool is in the queue, **When** the system generates a remediation proposal for it, **Then** the proposal targets only the system's improvement-loop knowledge artifacts (it does not modify tool source code or live tool definitions on its own), and the proposal cites the specific feedback and audit events it draws from.
3. **Given** an admin views a proposal, **When** they choose Accept, **Then** the proposal is applied to the improvement-loop knowledge artifact, the proposal status becomes "applied", and an audit event is recorded naming the admin, the tool, the proposal id, and the artifact change.
4. **Given** an admin views a proposal, **When** they choose Reject with a short rationale, **Then** the proposal status becomes "rejected" and the rationale is stored; future evaluation cycles do not re-propose the same change unless evidence materially changes.
5. **Given** a non-admin user is authenticated, **When** they request the admin review surface or any of its data, **Then** the system rejects the request with an authorization error.
6. **Given** an admin accepts a proposal, **When** subsequent dispatches of the same tool occur, **Then** the change influences orchestrator routing and tool-selection guidance for that tool (because it was applied to the existing improvement-loop knowledge artifact).

### User Story 3 - Defend the improvement loop against malicious feedback (Priority: P1)

User free-text feedback is untrusted input. A bad actor (or a careless user with an LLM-generated comment) might attempt to inject instructions designed to manipulate the system's improvement loop or to mislead the admin reviewer. The system must screen every piece of free-text feedback for prompt-injection patterns, jailbreak attempts, and social-engineering of the admin reviewer, before that text is allowed to influence any LLM context or be presented as guidance to a human.

**Why this priority**: P1, not optional. The improvement loop consumes user-supplied free text as input to an LLM. Without explicit screening, user feedback becomes an attack vector against the system's own behavior. Skipping this story ships a known-unsafe pipeline.

**Independent Test**: Submit feedback containing a representative set of injection / jailbreak / social-engineering payloads. Confirm each is detected, the feedback is recorded but its text is excluded from the improvement loop's input, the feedback is flagged in a quarantine queue, and the admin reviewer sees a clear "quarantined — do not act on text content" indicator.

**Acceptance Scenarios**:

1. **Given** a user submits feedback whose comment contains a recognized prompt-injection or jailbreak pattern, **When** the feedback is processed, **Then** the comment is marked quarantined, the feedback record retains the original raw text for audit, the comment text is excluded from the improvement-loop's LLM input, and the feedback is added to a quarantine review surface.
2. **Given** a user submits feedback whose comment contains content that attempts to address the admin reviewer directly (e.g., instructions or pleas aimed at a human), **When** the admin views the proposal generated from feedback aggregates, **Then** that comment text is not surfaced as guidance to the admin and is shown only with a clear "quarantined" label if shown at all.
3. **Given** a piece of feedback is quarantined, **When** the quality signal for the associated tool is computed, **Then** the sentiment and category from the quarantined record still count (a thumbs-down still counts) but the free text is excluded from any LLM-facing context.
4. **Given** the screening process produces a false positive, **When** an admin reviews the quarantine queue, **Then** the admin can release the feedback back into the improvement-loop input pool, and that release is itself audit-logged.

---

### User Story 4 - Retract or amend my own feedback (Priority: P2)

A user changes their mind about feedback they submitted (e.g., they realized the component was correct, or they want to edit their comment). They can retract or amend their own feedback within a retention window.

**Why this priority**: P2. Important for user trust and accuracy of the signal, but US-1 is shippable without it.

**Independent Test**: As a user, submit feedback, retract it, confirm the retracted feedback no longer contributes to quality signals, then submit new feedback and confirm it is counted.

**Acceptance Scenarios**:

1. **Given** the user has submitted feedback within the last 24 hours, **When** they retract it, **Then** the feedback is marked retracted, no longer contributes to any quality signal, but is preserved as an audit record.
2. **Given** a user retracts feedback, **When** they then submit fresh feedback for the same component, **Then** the new feedback is recorded as a new entry and is the one that counts.
3. **Given** a user's feedback was submitted more than 24 hours ago, **When** they attempt to retract or amend it, **Then** the system rejects the request with an "edit window expired" error.
4. **Given** a user attempts to retract or amend feedback belonging to another user, **When** the system processes the request, **Then** it rejects the request with an authorization error.

---

### Edge Cases

- A component is rendered but the user submits feedback after the conversation has ended. The feedback is still recorded against the original tool dispatch and conversation; the conversation having ended does not block submission.
- The same user opens the same component dispatch on multiple devices and submits feedback from both. The most recent submission wins; both submissions are preserved as history.
- A tool produces zero dispatches in the rolling window. It is treated as "insufficient data" and is not flagged as underperforming.
- A tool is freshly added and has only a handful of dispatches. The system requires a minimum dispatch count before the tool is eligible for "underperforming" classification (avoids false positives from small samples).
- The system's improvement loop is unavailable (e.g., the local model backing it is offline). New feedback continues to be recorded; quality signals continue to be computed; only the proposal generation pauses, and the admin review surface shows "improvement loop temporarily unavailable" without losing data.
- An admin attempts to accept a proposal that is no longer current (the underlying evidence changed). The system rejects the acceptance and surfaces the refreshed evidence; the admin must re-review.
- A component renders without an associated tool dispatch (e.g., a static layout or system message). The feedback control is still available, but the feedback is recorded against the rendering source rather than a tool, and is excluded from per-tool quality signals.
- The audit log retention horizon expires for an event that a feedback record references. The feedback record retains its own copy of the minimum identifying metadata so it remains interpretable; the link to the now-purged audit event is preserved by id but may be dangling.
- A user submits feedback on a component that was rendered for them by an admin's action (e.g., a system message). The feedback is still recorded against the user, not the admin, and influences tool quality the same way regular feedback does.
- Two admins simultaneously act on the same proposal. The first action wins; the second admin sees an updated state and a notice that the proposal was already actioned.

## Requirements *(mandatory)*

### Functional Requirements

#### Feedback capture (US-1)

- **FR-001**: The system MUST attach a stable per-dispatch identifier to every rendered component such that the user's feedback can be linked to the originating tool dispatch, agent, conversation, and user.
- **FR-002**: A user MUST be able to submit feedback on any rendered component without leaving the dashboard.
- **FR-003**: Feedback MUST capture sentiment as a binary value (positive / negative).
- **FR-004**: Feedback MUST allow the user to optionally select one category from a fixed taxonomy: `wrong-data`, `irrelevant`, `layout-broken`, `too-slow`, `other`. If no category is selected, the system records `unspecified`.
- **FR-005**: Feedback MUST allow the user to optionally include a free-text comment.
- **FR-006**: The system MUST acknowledge receipt of feedback to the user within one second under normal conditions.
- **FR-007**: Feedback MUST be persisted to a per-user, append-only store and MUST be queryable by user, by tool, by agent, by category, by sentiment, and by time window.
- **FR-008**: Each feedback submission MUST produce a corresponding audit-log entry attributed to the submitting user.
- **FR-009**: A user MUST NOT be able to read, retract, amend, or otherwise act on another user's feedback. Cross-user attempts MUST be rejected with an authorization error.
- **FR-009a**: The system MUST apply a per-user-per-component dedup window (default 10 seconds, operator-configurable) to feedback submissions. A new submission from the same user against the same component-dispatch within the window MUST update the existing in-window record in place (latest values win) rather than create a new record, and MUST NOT emit a new audit-log row for the duplicate. Submissions outside the window create a new record that supersedes prior records per FR-007 (which DO emit audit rows).

#### Underperforming-tool detection (US-2)

- **FR-010**: The system MUST compute a per-tool quality signal on a 14-day rolling window. The signal MUST combine audit-log dispatch failure rate AND negative-feedback rate. Both the window length and the threshold values below MUST be operator-configurable, with the values stated here as defaults.
- **FR-011**: A tool MUST become eligible for "underperforming" classification only after at least 25 dispatches within the rolling window (default minimum dispatch count), to avoid small-sample false positives.
- **FR-012**: When an eligible tool's failure rate is ≥ 20% OR its negative-feedback rate is ≥ 30% over the rolling window, the system MUST mark it underperforming and surface it in the admin review queue.
- **FR-012a**: On each first transition of a tool from any non-underperforming state into "underperforming," the system MUST emit an audit-log event of type `tool_flagged` identifying the tool, the agent, the window, and the metric values that triggered the flag. Repeat-stays in the underperforming state across cycles MUST NOT emit new `tool_flagged` events; a separate `tool_recovered` event MUST be emitted when a previously-flagged tool re-enters a non-underperforming state.
- **FR-012b**: The admin review surface MUST display a passive indicator (badge with the count of currently-pending underperforming-tool flags). No out-of-band notifications (email, push, Slack) are required by this feature; the audit-log `tool_flagged` event exists so any future out-of-band alerting can be added without redesigning detection.
- **FR-013**: The admin review queue MUST be accessible only to users holding the `admin` role. Non-admin access MUST return an authorization error.
- **FR-014**: The admin review queue MUST present, per underperforming tool, the supporting evidence: dispatch count, failure count, negative feedback count, category breakdown, and references to underlying audit events and feedback records.

#### Improvement-loop integration (US-2 — applied side)

- **FR-015**: User feedback MUST be made available as an additional input source to the system's existing tool-improvement loop, alongside the loop's existing tool-outcome signals.
- **FR-016**: The system MUST generate a remediation proposal per underperforming tool. A proposal MUST take the form of a change to the improvement loop's existing knowledge artifacts. A proposal MUST NOT modify tool source code, schemas, or live tool definitions on its own.
- **FR-017**: An admin MUST be able to Accept, Reject, or Edit-and-Accept a proposal. Each action MUST be recorded in the audit log including actor, tool, proposal id, decision, optional rationale, and resulting artifact change.
- **FR-018**: An accepted proposal MUST be applied to the improvement-loop's existing knowledge artifact such that subsequent dispatches of the relevant tool are influenced by the change.
- **FR-019**: A rejected proposal with a stored rationale MUST NOT be re-proposed on the next cycle unless the supporting evidence has materially changed.
- **FR-020**: If the improvement-loop service is unavailable, feedback capture and quality-signal computation MUST continue uninterrupted; only proposal generation MUST pause and surface a clear status to admins.

#### Safety: prompt-injection / nefarious-behavior screening (US-3)

- **FR-021**: All free-text feedback content MUST be screened for prompt-injection patterns, jailbreak attempts, and social-engineering of the admin reviewer BEFORE that content reaches any LLM context or is surfaced as guidance to a human reviewer. Screening MUST happen at two points (defense in depth):
  - **Inline at submit time**, so the submitting user receives immediate acknowledgement and the record is marked clean or quarantined before it is durably stored as "active" input for any downstream consumer.
  - **As a pre-pass inside the improvement loop**, immediately before any synthesizer prompt is constructed, applied to every record entering that cycle's input pool (including records previously released from quarantine and records whose original screening predated a rule update).
- **FR-021a**: When the inline screen and the loop pre-pass disagree (e.g., loop pre-pass flags a record the inline pass cleared, due to tightened rules), the stricter result wins; the record is moved to quarantine and the inconsistency is recorded for screening-accuracy review.
- **FR-022**: Feedback whose text is flagged by screening MUST be marked quarantined. The original raw text MUST still be retained on the feedback record for audit purposes.
- **FR-023**: Quarantined feedback text MUST be excluded from the improvement loop's LLM input.
- **FR-024**: Quarantined feedback's sentiment and category MUST still count toward quality signals (a quarantined thumbs-down is still a thumbs-down).
- **FR-025**: When feedback content is presented in any LLM context (e.g., the improvement-loop synthesizer), it MUST be presented as escaped, clearly-labeled untrusted data — never as instructions.
- **FR-026**: The system MUST provide a quarantine review surface for admins. An admin MUST be able to release a quarantined item back into the improvement-loop input pool. Releases MUST be audit-logged.
- **FR-027**: Quarantine outcomes (flag vs. clear, release vs. hold) MUST be auditable and queryable for ongoing screening accuracy review.

#### Retraction and amendment (US-4)

- **FR-028**: A user MUST be able to retract or amend their own feedback within 24 hours of the original submission timestamp. After 24 hours the record is locked and retract/amend requests MUST be rejected with a clear "edit window expired" error. After retraction (within the window), the feedback MUST NOT count toward any quality signal but MUST remain as an audit record.
- **FR-029**: Amendment MUST be modeled as a new entry that supersedes the prior; both versions MUST be retained for audit. Amendment is permitted only within the same 24-hour window as retraction (FR-028).

#### Cross-cutting

- **FR-030**: All operations described in this spec — submit, view, accept, reject, retract, amend, release-from-quarantine — MUST produce audit-log entries with the actor's identity.
- **FR-031**: Per-user isolation MUST be tested end-to-end: there MUST be zero cross-user feedback leaks.
- **FR-032**: Feedback retention MUST follow the project's existing audit-data retention policy unless an explicit override is configured.

### Key Entities *(include if feature involves data)*

- **ComponentFeedback**: A single feedback submission. Identifies which user, which conversation, which tool dispatch, sentiment, category (from the fixed taxonomy or `unspecified`), the raw free-text comment if any, the safety status (`clean` or `quarantined`), the safety reason if quarantined, the lifecycle status (`active`, `superseded`, `retracted`), and the time of submission. Append-only with new entries superseding old by reference.
- **ToolQualitySignal**: A periodic computed snapshot per tool over a rolling window. Identifies tool, agent, window start, window end, total dispatch count, failure count, negative feedback count, computed score, and current status (`healthy` | `insufficient-data` | `underperforming`).
- **KnowledgeUpdateProposal**: A system-generated proposed change to the improvement-loop's knowledge artifact, scoped to one underperforming tool. Identifies the tool, agent, generation time, the proposed change as a diff against the artifact, the supporting evidence as references to audit events and feedback records, lifecycle status (`pending`, `accepted`, `applied`, `rejected`, `superseded`), reviewer identity if reviewed, review time, and reviewer rationale if rejected.
- **QuarantineEntry**: A pointer to a ComponentFeedback whose text was flagged. Carries the screening reason, screening confidence/category, current status (`held`, `released`, `dismissed`), and the admin who actioned it if any.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can submit feedback on a rendered component and see acknowledgement in under 1 second on a normal connection in 95% of attempts.
- **SC-002**: At least 99% of submitted feedback events are durably recorded — there is at most 1 lost event per 100 submissions, even across brief network interruptions.
- **SC-003**: When a tool's underlying outcome and feedback evidence crosses the configured threshold, the tool is surfaced to the admin review queue within one evaluation cycle (≤ 24 hours).
- **SC-004**: Cross-user feedback leakage is exactly zero across all end-to-end isolation tests (mirrors feature 003 audit-log isolation guarantee).
- **SC-005**: Of feedback submissions whose free-text content is a known-malicious test payload, ≥ 99% are flagged and quarantined; their text content does not appear in any LLM-facing context produced by the improvement loop.
- **SC-006**: Admin time to identify a flagged underperforming tool drops at least 50% relative to the current path (manually scanning audit data).
- **SC-007**: For tools where the admin has accepted at least one improvement proposal, the proportion of negative feedback in the next equivalent rolling window decreases for at least 60% of those tools (measured at the tool population level, not per-tool deterministically).
- **SC-008**: The improvement loop's behavior remains unchanged in the absence of feedback: existing tool-outcome-driven improvements are not regressed.

## Assumptions

- "Developer" in the user request is interpreted as the existing `admin` role in the project's identity system. No new role is introduced by this feature.
- Per-user isolation for feedback follows the same model as the project's audit log feature: each record is owned by exactly one user; cross-user reads return empty / 404 indistinguishably from non-existence.
- The "system that already improves tools" refers to the existing background improvement loop in this project. This feature extends that loop's input set rather than introducing a parallel improvement system.
- Tool source code is not modified at runtime by automated processes. The auto-update mechanism operates on knowledge artifacts the existing improvement loop already maintains, and only after admin acceptance.
- Feedback retention defaults to the project's audit retention policy unless explicitly overridden.
- Components that do not originate from a tool dispatch (e.g., static layout, system messages) may still receive feedback; that feedback is recorded for audit but does not influence per-tool quality signals.
